2026-06-17

1. 트랜잭션

데이터베이스의 상태를 변화시키기 위해 수행하는 하나의 논리적 작업 단위(또는 일련의 연산)
중간에 일부만 반영되면 데이터가 깨질 수 있으므로, 전부 반영되거나 전혀 반영되지 않는 단위로 묶어 처리함

특징 (ACID)

  • 원자성(Atomicity): 트랜잭션의 연산이 모두 성공적으로 반영되거나, 아예 하나도 반영되지 않아야 함
  • 일관성(Consistency): 트랜잭션이 끝난 뒤에도 데이터베이스는 정해진 무결성 제약을 만족해야 함
  • 독립성/고립성(Isolation): 수행 중인 트랜잭션의 중간 결과에 다른 트랜잭션이 끼어들 수 없어야 함
  • 영속성(Durability): 완료된 트랜잭션의 결과는 시스템 장애가 일어나도 사라지지 않고 영구적으로 반영됨

TCL 명령어

원자성을 보장하려면 트랜잭션의 시작·종료·되돌리기를 명시적으로 제어해야 함
이를 위해 사용하는 SQL 명령을 TCL(Transaction Control Language) 이라고 부름

  • COMMIT: 트랜잭션이 정상적으로 종료되었음을 알리고, 변경 사항을 데이터베이스에 영구 반영
  • ROLLBACK: 트랜잭션이 비정상적으로 종료되어, 그동안의 변경 사항을 취소하고 이전 상태로 되돌림
  • SAVEPOINT: 트랜잭션 중간에 특정 지점을 표시해두는 명령
    ROLLBACK과 함께 쓰면 트랜잭션 전체가 아니라 표시한 지점까지만 되돌릴 수 있음

트랜잭션의 상태 

                     ┌──────────────┐
                     │   활성화      │
                     │  (Active)    │
                     └──────┬───────┘
                            │ 마지막 연산 수행
                            ▼
                  ┌──────────────────┐
        ┌────────│   부분 완료       │────────┐
        │        │(Partially Done)  │        │
        │        └──────────────────┘        │
        │ 오류                                │ COMMIT
        ▼                                    ▼
   ┌──────────┐                       ┌──────────┐
   │  실패     │                      │  완료     │
   │ (Failed) │                       │(Committed)│
   └────┬─────┘                       └──────────┘
        │ ROLLBACK
        ▼
   ┌──────────┐
   │  철회     │
   │(Aborted) │
   └──────────┘
  • 활성화(Active): 트랜잭션이 시작되어 연산을 처리 중인 상태
  • 부분 완료(Partially Committed): 마지막 연산까지 끝났지만 아직 결과가 DB에 반영되지 않은 상태
  • 완료(Committed): 결과가 DB에 정상적으로 반영된 상태
  • 실패(Failed): 처리 도중 오류가 발생해 트랜잭션이 중단된 상태
  • 철회(Aborted): 실패 후 ROLLBACK을 수행해 데이터베이스가 이전 상태로 되돌아간 상태

cf) 트랜잭션 격리 수준 (Isolation Level)

여러 트랜잭션이 동시에 실행될 때, 서로의 작업을 얼마나 격리할 것인지 정하는 수준
격리 수준이 낮을수록 동시성은 높아지지만 데이터 일관성 문제가 발생할 수 있음

  • Read Uncommitted: 다른 트랜잭션이 COMMIT하지 않은 변경 사항까지도 조회 가능
    동시성은 가장 높지만 Dirty Read가 발생할 수 있음
  • Read Committed: COMMIT된 데이터만 다른 트랜잭션이 조회할 수 있음
    Dirty Read는 막지만, 같은 데이터를 두 번 읽었을 때 값이 달라지는 Non-Repeatable Read 발생 가능
  • Repeatable Read: 트랜잭션이 한 번 읽은 데이터는 다른 트랜잭션이 갱신·삭제할 수 없음
    단, 새로운 행이 추가되는 것은 막지 못해 Phantom Read가 발생할 수 있음
  • Serializable: 트랜잭션이 읽은 데이터에 대해 다른 트랜잭션이 갱신·삭제·삽입을 모두 할 수 없음
    가장 안전하지만 동시성이 가장 낮아 성능이 떨어짐

동시성과 고립성은 상충 관계에 있음
동시성이 높으면 처리량은 올라가지만 데이터 일관성 문제가 발생할 위험이 커지고,
고립성을 높이면 일관성은 보장되지만 다른 트랜잭션이 대기해야 하므로 효율이 떨어짐

2. 락(Lock)

트랜잭션이 처리되는 순서를 보장하고, 같은 데이터에 동시에 접근하지 못하도록 막는 메커니즘
운영체제의 상호 배제(Mutual Exclusion) 기법과 유사한 원리

락의 종류

  • 공유 락(Shared Lock, 읽기 락): 데이터를 읽기 위한 락
    읽기는 일관성에 영향을 주지 않으므로, 동일 데이터에 여러 공유 락이 동시에 걸릴 수 있음
  • 배타 락(Exclusive Lock, 쓰기 락): 데이터를 수정하기 위한 락
    데이터에 하나의 배타 락이 걸려 있는 동안에는 다른 어떤 락(공유든 배타든)도 걸 수 없음

트랜잭션의 교착 상태(Deadlock)

여러 트랜잭션이 서로가 가진 락을 기다리며 무한 대기에 빠진 상태
한 트랜잭션이 어떤 데이터의 락을 보유한 상태에서 또 다른 데이터의 락을 요청하고, 그 데이터의 락은 또 다른 트랜잭션이 보유하고 있는 상황이 맞물려 발생함

해결 방법

1. 예방 기법

트랜잭션을 시작하기 전에 필요한 모든 데이터의 락을 한 번에 미리 획득하는 방식
하나라도 못 얻으면 트랜잭션을 시작하지 않으므로 교착 상태가 원천적으로 발생하지 않음
단, 락을 오래 보유하게 되어 동시성이 떨어짐

2. 회피 기법

트랜잭션이 들어온 순서(타임스탬프)에 따라 교착 상태를 피하는 방식

  • wait-die: 락을 요청하려는 트랜잭션이 락 보유 트랜잭션보다 오래되었다면 락을 얻을 때까지 대기
    더 최신이라면 락 획득을 포기하고 자신을 종료 (die)
  • wound-wait: 락을 요청하려는 트랜잭션이 락 보유 트랜잭션보다 오래되었다면 보유자를 강제로 종료시키고 락을 빼앗음 (wound)
    더 최신이라면 락이 풀릴 때까지 대기

두 방식 모두 “오래된 트랜잭션에게 우선권을 준다"는 원칙은 같지만, 양보의 방향이 반대임

3. 이상(Anomaly)

테이블이 잘 정규화되지 않아 속성 간 종속이나 데이터 중복이 있을 때 발생하는 문제
같은 정보가 여러 곳에 흩어져 있을 때 데이터의 정확성을 해치게 됨

  • 삽입 이상(Insertion Anomaly): 새 데이터를 넣고 싶을 때 의도치 않게 다른 데이터까지 함께 넣어야 하는 문제
  • 갱신 이상(Update Anomaly): 데이터를 수정할 때 중복된 모든 위치를 동시에 갱신하지 못해 데이터 모순이 발생하는 문제
  • 삭제 이상(Deletion Anomaly): 일부 데이터를 삭제하려 했을 때 의도치 않은 다른 정보까지 함께 사라지는 문제

4. 정규화

이상 현상을 해결하기 위해 테이블을 더 작은 테이블로 분해하는 과정
분해 단계에 따라 단계별 “정규형"이 정의되며, 보통 제3정규형(또는 BCNF)까지 진행하는 것이 일반적임
정규화 단계가 높아질수록 데이터 중복은 줄지만 조인 비용이 늘기 때문에, 상황에 맞게 적정 수준을 선택해야 함

정규화 단계 

   비정규형
      │  ① 모든 속성을 원자 값으로 분해
      ▼
   제1정규형 (1NF)
      │  ② 부분 함수 종속 제거
      ▼
   제2정규형 (2NF)
      │  ③ 이행적 함수 종속 제거
      ▼
   제3정규형 (3NF)
      │  ④ 모든 결정자가 후보 키가 되도록 분해
      ▼
   보이스-코드 정규형 (BCNF)
      │  ⑤ 다치 종속 제거
      ▼
   제4정규형 (4NF)
      │  ⑥ 조인 종속 제거
      ▼
   제5정규형 (5NF)

제1정규형 (1NF)

테이블의 모든 속성 값이 더 이상 분해될 수 없는 원자 값(atomic value) 으로 구성되어야 함

[ 위반 — 한 셀에 여러 값 ]
| 회원ID | 취미              |
| ------ | ----------------- |
| 1      | 독서, 영화, 등산  |

[ 1NF ]
| 회원ID | 취미   |
| ------ | ------ |
| 1      | 독서   |
| 1      | 영화   |
| 1      | 등산   |

제2정규형 (2NF)

1NF를 만족하면서, 부분 함수 종속을 제거해 모든 일반 속성이 기본 키 전체에 종속되어야 함
즉, 기본 키의 진부분집합이 일반 속성을 결정해서는 안 됨

[ 위반 — (학생ID, 과목ID)가 키인데 과목명은 과목ID에만 종속 ]
| 학생ID | 과목ID | 과목명 | 점수 |
| ------ | ------ | ------ | ---- |
| 1      | A01    | 자료구조 | 95 |
| 2      | A01    | 자료구조 | 88 |

[ 2NF — 테이블 분해 ]
[수강]: 학생ID, 과목ID, 점수
[과목]: 과목ID, 과목명

cf) 진부분집합: 집합 A가 집합 B의 부분집합이면서 B와 같지 않은 경우

제3정규형 (3NF)

2NF를 만족하면서, 이행적 함수 종속을 제거해야 함
즉, “A → B, B → C이면 A → C"처럼 키가 아닌 속성이 다른 일반 속성을 결정하는 관계가 없어야 함

[ 위반 — 회원ID → 등급ID → 등급명 (이행적 종속) ]
| 회원ID | 등급ID | 등급명 |

[ 3NF ]
[회원]: 회원ID, 등급ID
[등급]: 등급ID, 등급명

보이스-코드 정규형 (BCNF)

“강한 제3정규형"이라고도 함
3NF를 만족하면서, 모든 결정자가 후보 키가 되도록 테이블을 분해해야 함
3NF만으로는 잡히지 않는 일부 이상 현상까지 제거할 수 있음

제4정규형 (4NF)

BCNF를 만족하면서, 다치 종속(multi-valued dependency) 을 제거해야 함

cf) 다치 종속: 한 속성이 다른 여러 속성의 값의 집합을 결정하는 관계
예: 한 학생이 여러 과목을 듣고 여러 동아리에 속할 때, 과목과 동아리가 서로 독립이라면 한 테이블로 묶지 말아야 함

제5정규형 (5NF)

4NF를 만족하면서, 조인 종속이 없어야 함
테이블을 더 작게 분해한 뒤 다시 조인했을 때 원래 테이블이 손실 없이 복원되도록 분해해야 함

역정규화 (Denormalization)

정규화된 테이블을 의도적으로 낮은 정규화 단계로 되돌리는 작업
조인 비용을 줄여 읽기 성능을 끌어올리고 싶을 때 사용함
대신 데이터 중복으로 인한 이상 현상의 위험을 다시 떠안게 되므로, 트레이드오프를 신중히 따져야 함
