1. REST(Representational State Transfer)

HTTP 통신을 효율적으로 활용하기 위해 고안된 아키텍처
인터넷상의 자원을 URI로 식별하고, 그 자원에 대해 HTTP 메서드로 CRUD 연산을 수행함

이름의 의미를 풀어 보면

  • Representational(표현): 자원 자체가 아니라, 자원의 상태를 표현한 데이터(JSON, XML 등)를 주고받음
  • State Transfer(상태 전이): 클라이언트가 요청한 자원의 현재 상태를 응답으로 전달함

특징

  • 일관된 인터페이스: URI로 자원을 가리키고, HTTP 메서드로 동작을 표현하는 일관된 규약을 사용함
  • 클라이언트-서버 구조: 요청을 보내는 클라이언트와 자원을 보유한 서버가 명확히 분리되어 있어, 양쪽이 서로 독립적으로 발전할 수 있음
  • 무상태성: 서버는 클라이언트의 이전 요청 상태를 저장하지 않음
    각 요청은 그 자체로 완결되어야 하며, 인증 정보나 로그인 상태 같은 컨텍스트는 클라이언트가 관리함
  • 캐시 가능성: HTTP 표준을 그대로 활용하기 때문에 응답을 캐싱해 재사용할 수 있음
    서버 부하를 줄이고 클라이언트는 더 빠르게 응답을 받을 수 있음
  • 자체 표현 구조: 메시지가 자원, 행위, 표현으로 구성되어 있어 요청 자체를 보면 의도를 파악할 수 있음
  • 계층형 구조: 보안, 암호화, 로드 밸런싱 등 기능을 별도 계층으로 끼워 넣을 수 있어 서버 구조를 유연하게 확장할 수 있음

HTTP 기반이라 별도의 인프라가 필요 없다는 장점이 있는 반면, 자원에 대한 연산을 HTTP 메서드로만 처리해야 하므로 표현 가능한 동작이 한정적이라는 한계도 있음

cf) URI, URL, URN

  • URI (Uniform Resource Identifier): 자원을 식별하는 모든 문자열의 통칭
  • URL (Uniform Resource Locator): 자원이 “어디에” 있는지 위치를 나타내는 식별자
  • URN (Uniform Resource Name): 자원의 “이름"으로 식별하는 식별자 (위치와 무관)
                 ┌──────────────────────────────┐
                 │             URI              │
                 │  ┌─────────┐    ┌─────────┐  │
                 │  │   URL   │    │   URN   │  │
                 │  └─────────┘    └─────────┘  │
                 └──────────────────────────────┘

   URL 예시: https://example.com/posts/123
   URN 예시: urn:isbn:978-0-13-468599-1

URL과 URN은 모두 URI의 한 종류이며, 일상적으로는 URI ≈ URL로 통용되는 경우가 많음

2. REST API

REST 아키텍처 스타일을 따라 설계된 API
요청은 다음 세 가지 요소로 구성됨

  • 자원(Resource) — URI로 식별
  • 행위(Verb) — HTTP 메서드(GET, POST, PUT, DELETE 등)로 표현
  • 표현(Representation) — JSON, XML 등의 형식으로 주고받는 데이터

REST API 동작 방식 

   [클라이언트]                              [서버]
       |                                       |
       | ── ① URI + HTTP 메서드로 요청 ─────────▶ |
       |     (예: GET /users/1)                |
       |                                       |
       |                                       │ ② 요청 해석 및 처리
       |                                       │   (자원 조회·생성·수정·삭제)
       |                                       |
       |  ◀───── ③ HTTP 응답(상태 코드 + 데이터) ── │
       |                                       |
       │   ④ 응답 수신 및 활용                    |
       │   (JSON/XML 등을 파싱)                  |
       |                                       |
  1. 클라이언트가 URI로 식별한 자원에 대해 HTTP 메서드를 사용해 요청을 보냄
  2. 서버는 수신한 HTTP 요청 메시지를 해석하고, 그에 맞는 처리를 수행함
  3. 서버는 처리 결과를 HTTP 응답으로 반환함 (요청의 성공 여부를 나타내는 상태 코드와 자원 정보를 포함)
  4. 클라이언트는 JSON/XML 등의 형태로 전달된 응답을 수신해 활용함

cf) RESTful하게 API를 설계하기 위한 규칙

  • 자원에 대한 행위는 HTTP 메서드로 나타내며, HTTP 메서드나 동작에 해당하는 단어가 URI에 포함되어서는 안 됨
    (예: GET /getUser ❌ → GET /users ⭕)
  • HTTP 메서드는 명시적으로 사용되어야 함
    POST 메서드로 생성(Create)뿐만 아니라 수정(Update)까지 처리한다면 의도를 명시적으로 드러낸다고 볼 수 없음
  • URI 경로는 슬래시(/)로 자원의 계층 관계를 표현하며, 경로 끝에 슬래시를 붙이지 않음
    (예: /users/1/posts ⭕, /users/1/posts/ ❌)
  • URI 경로에는 언더바(_) 대신 하이픈(-)을 사용하고, 가독성을 위해 소문자로 표기하는 것을 권장함

3. HTTP 메서드

자원에 어떤 동작을 수행할지를 나타내는 식별자
REST에서는 다음 네 가지 메서드가 자원에 대한 CRUD 연산과 직접 대응됨

  • POST: 자원을 새로 생성함
    요청 바디에 생성할 데이터를 담아 보내며, 같은 요청을 반복하면 자원이 매번 새로 만들어질 수 있어 멱등하지 않음
  • GET: 자원을 조회함
    서버의 상태를 변경하지 않는 안전한 메서드이며, 같은 요청을 여러 번 보내도 결과가 동일해 멱등함
  • PUT: 자원을 전체 교체함 (없으면 생성)
    동일한 요청을 여러 번 보내도 최종 상태가 같으므로 멱등함
  • DELETE: 자원을 삭제함
    이미 삭제된 자원에 다시 요청을 보내도 최종 상태가 같아 멱등함

CRUD 연산과 HTTP 메서드 매핑

CRUD 연산 HTTP 메서드 예시
Create (생성) POST POST /users
Read (조회) GET GET /users/1
Update (수정) PUT / PATCH PUT /users/1
Delete (삭제) DELETE DELETE /users/1

멱등성(Idempotency): 같은 요청을 여러 번 보내도 서버의 상태가 동일하게 유지되는 성질
안전성(Safety): 요청이 서버의 상태를 변경하지 않는 성질 (GET, HEAD 등이 해당)

cf) 기타 메서드

  • PATCH: 자원의 일부 필드만 부분적으로 수정함
    PUT이 자원 전체를 교체하는 데 비해, PATCH는 변경할 필드만 보냄
  • HEAD: GET과 동일한 요청을 보내되 응답 바디 없이 헤더만 받음
    자원의 존재 여부나 메타데이터(크기, 수정일 등)를 가볍게 확인할 때 사용
  • TRACE: 요청이 서버까지 도달하는 동안 거친 경로를 그대로 되돌려 받음
    주로 디버깅 용도로 쓰이며, 보안상의 이유로 운영 서버에서는 비활성화하는 경우가 많음
  • CONNECT: 클라이언트와 특정 서버 사이에 양방향 터널을 설정함
    주로 HTTPS 프록시에서 SSL/TLS 터널링에 사용
  • OPTIONS: 서버가 지원하는 메서드 목록을 확인할 때 사용
    브라우저가 CORS preflight 요청을 보낼 때도 이 메서드를 사용함
