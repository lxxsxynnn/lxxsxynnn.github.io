[CS 스터디] 관계형 데이터베이스

2026-06-16

1. 키

테이블의 수많은 튜플 중에서 특정 튜플을 식별하기 위한 속성(또는 속성의 집합)
키는 다음 두 가지 특성을 기준으로 분류함

  • 유일성(Uniqueness): 키 값 하나로 튜플 한 행을 유일하게 식별할 수 있는 성질
  • 최소성(Minimality): 식별에 꼭 필요한 속성만으로 키를 구성하는 성질

키의 종류

  • 슈퍼 키(Super Key): 유일성은 만족하지만 최소성은 만족하지 않을 수 있는 키
    예: (회원ID, 이름) — 회원ID만으로도 식별 가능하지만 이름까지 묶어도 슈퍼 키
  • 후보 키(Candidate Key): 유일성과 최소성을 모두 만족하는 키
    하나의 테이블에 여러 개 존재할 수 있음 (예: 회원ID, 이메일)
  • 기본 키(Primary Key): 여러 후보 키 중 대표로 선택된 키
    NULL 값을 가질 수 없고, 중복도 허용되지 않음
  • 대체 키(Alternate Key): 후보 키 중 기본 키로 선택되지 않은 나머지 키
    여전히 유일성을 보장하므로 보조적인 식별자로 활용할 수 있음
  • 외래 키(Foreign Key): 다른 테이블의 기본 키를 참조해 두 테이블을 연결하는 키
    관계형 모델에서 테이블 간 관계를 표현하는 핵심 수단

2. 무결성

데이터베이스에 저장된 값이 실제 사실과 일치하고(정확성), 항상 일정하게 유지되도록(일관성) 보장하는 성질
잘못된 입력이 들어와도 데이터의 신뢰성이 깨지지 않도록 DBMS가 강제하는 제약 조건

종류

  • 개체 무결성(Entity Integrity): 모든 테이블은 기본 키를 가져야 하며, 그 값은 NULL이거나 중복될 수 없음
    → 각 행이 유일하게 식별 가능해야 한다는 원칙
  • 도메인 무결성(Domain Integrity): 속성 값은 그 속성에 허용된 도메인(값의 집합) 안에 있어야 함
    예: 나이 컬럼은 정수, 성별 컬럼은 ‘M’/‘F’ 중 하나만 허용
  • 참조 무결성(Referential Integrity): 외래 키의 값은 참조 대상 테이블의 기본 키 값과 일치하거나 NULL이어야 함
    → 존재하지 않는 행을 참조하는 “고아 데이터"가 생기지 않도록 막아 줌

3. 인덱스

테이블의 데이터를 빠르게 찾기 위해, 속성 값과 해당 튜플의 위치(주소)를 별도로 정리해둔 자료구조
책 뒤에 있는 색인(index)처럼, 전체를 훑지 않고도 원하는 데이터로 바로 점프할 수 있도록 도와줌

   [ 원본 테이블 ]                     [ 이름 컬럼 인덱스 ]
   ┌─────┬──────┬──────┐               ┌──────┬─────────┐
   │ id  │ name │ age  │               │ name │ row_ptr │
   ├─────┼──────┼──────┤               ├──────┼─────────┤
   │ 101 │ 영희  │  29  │               │ 미나  │  → 103  │
   │ 102 │ 철수  │  31  │               │ 영희  │  → 101  │
   │ 103 │ 미나  │  27  │               │ 지훈  │  → 104  │
   │ 104 │ 지훈  │  35  │               │ 철수  │  → 102  │
   └─────┴──────┴──────┘               └──────┴─────────┘
                                       (정렬된 상태로 보관)

장점

  • 인덱스가 정렬된 상태로 유지되므로 검색 속도가 크게 빨라짐
    특히 데이터가 많을수록 풀 스캔과의 격차가 벌어짐

단점

  • 인덱스를 저장하기 위한 추가 공간이 필요함
  • 데이터의 삽입·수정·삭제 시 인덱스도 함께 갱신해야 하므로 쓰기 성능이 저하됨
  • 삭제 연산이 일어나도 인덱스 항목은 즉시 물리적으로 제거되지 않고, 일단 “사용하지 않음"으로 표시된 채 남아 있다가 나중에 일괄적으로 정리되는 경우가 많음 (lazy deletion)
   [ '미나' 삭제 직후 — 인덱스에는 흔적이 남아 있음 ]

   원본 테이블                            이름 인덱스
   ┌─────┬──────┬──────┐                 ┌──────┬───────────┐
   │ 101 │ 영희  │  29  │                 │ 미나  │✗ (삭제 표시)│
   │ 102 │ 철수  │  31  │                 │ 영희  │   → 101   │
   │ 104 │ 지훈  │  35  │                 │ 지훈  │   → 104   │
   └─────┴──────┴──────┘                 │ 철수  │   → 102   │
                                         └──────┴───────────┘

인덱스는 대표적으로 해시 테이블과 B-트리 / B+ 트리 두 가지 방식으로 구현됨

해시 테이블 인덱스

  • 속성 값을 해시 함수에 통과시켜 나온 해시 값으로 인덱싱하는 방식
  • 정확히 일치하는 값으로 검색할 때만 사용 가능 (예: WHERE name = '영희')
  • 부등호(>, <) 비교나 부분 일치(LIKE) 같은 범위 검색에는 활용할 수 없음
  • 속도는 매우 빠르지만 활용 범위가 좁아 일반적인 RDBMS의 기본 인덱스로는 잘 쓰이지 않음

B-트리 / B+ 트리 인덱스

  • 인덱스를 정렬된 트리 형태로 관리하는 방식
  • 삽입·삭제 시에도 균형이 유지되므로 삽입·갱신·삭제 비용이 안정적
  • 범위 검색이 가능해 BETWEEN, 부등호 비교, 부분 일치 등에 활용할 수 있음
    특히 B+ 트리는 리프 노드들이 연결 리스트로 이어져 있어 범위 스캔에 매우 유리함
  • 위와 같은 이유로 대부분의 관계형 DBMS는 B+ 트리를 기본 인덱스 구조로 사용함

4. ORM (Object-Relational Mapping)

객체 지향 언어의 객체와 관계형 데이터베이스의 테이블을 자동으로 매핑해주는 도구
SQL을 직접 작성하지 않아도 객체를 다루듯 데이터베이스에 접근할 수 있어, 객체 지향 코드와 DB 사이의 패러다임 차이를 줄여 줌

장점

  • SQL 대신 객체 지향적인 코드로 데이터 조작을 표현할 수 있어 가독성이 좋아짐
  • DBMS 종류가 바뀌어도 ORM이 SQL 차이를 흡수해주므로 이식성이 높아짐
  • 반복적인 CRUD 코드를 자동화해 개발 생산성이 향상됨

단점

  • 추상화 계층이 한 단계 더 끼는 만큼 성능 튜닝이 어렵고, 복잡한 쿼리는 직접 SQL을 작성하는 편이 나을 수 있음
  • 연관 객체를 반복 조회할 때 발생하는 N+1 문제 등, ORM 특유의 함정을 이해하고 있어야 함

대표 예시

  • Java: Hibernate, JPA
  • Python: Django ORM, SQLAlchemy
  • Ruby: ActiveRecord
  • Node.js: Sequelize, Prisma
