[CS 스터디] 가상 메모리

2026-06-04

1. 가상 메모리 (Virtual Memory)

물리적 메모리(RAM)의 한계를 극복하기 위해, 디스크의 일부 영역(백킹 스토어)을 메모리처럼 활용하여 프로세스를 실행하는 기술

  • 특징: 사용자나 프로세스에게는 프로세스 전체가 메모리에 로드된 것처럼 보이지만, 실제로는 당장 실행에 필요한 일부만 메모리에 올라가 있음
  • 장점:
    • 프로그램이 물리적 메모리 크기에 대한 제약을 덜 받음 (메모리보다 큰 프로그램 실행 가능)
    • 동시에 더 많은 프로그램을 실행할 수 있어 CPU 이용률과 처리율(Throughput) 향상
    • 필요한 영역만 로드하므로 전체 프로세스를 스와핑하는 횟수가 줄어들어 I/O 오버헤드 감소

2. 요구 페이징 (Demand Paging)

프로세스 실행 시 모든 페이지를 한 번에 메모리에 올리지 않고, 실제로 요청이 들어온(필요한) 페이지만 메모리에 로드하는 방식

  • 유효-무효 비트 (Valid-Invalid Bit): 페이지 테이블에 존재하며, 해당 페이지가 현재 물리 메모리에 있는지 여부를 나타냄
    • Valid: 페이지가 물리 메모리에 로드되어 있음
    • Invalid: 페이지가 아직 물리 메모리에 없거나(디스크에 존재), 유효하지 않은 주소 영역임

3. 페이지 폴트 (Page Fault)

CPU가 참조하려는 페이지가 물리 메모리에 없는 상황 (페이지 테이블의 비트가 ‘Invalid’인 경우)

페이지 폴트 발생 시 처리 과정 도식화 

    [ CPU ] ── (1) 논리 주소 참조 ──> [ 페이지 테이블 ]
                                      (Valid/Invalid 비트 확인)
                                           │
                        (2) Invalid (페이지 폴트 발생 - Trap)
                                           ↓
                                     [ 운영체제 (OS) ]
                                           │ (3) 보호 영역 접근인지 확인 후,
                                           │     디스크에서 해당 페이지 탐색
                                           ↓
                                [ 백킹 스토어 (디스크 영역) ]
                                           │ (4) 물리 메모리의 빈 프레임에
                                           │     해당 페이지를 로드 (Swap-in)
                                           ↓
                                     [ 물리 메모리 ]
                                           │ (5) 페이지 테이블 업데이트
                                           │     (프레임 번호 기록 및 Valid 비트로 변경)
                                           ↓
    [ CPU ] <── (6) 명령 재시작 ────── [ 페이지 테이블 ]
  1. 주소 참조: CPU가 프로세스를 실행하며 페이지 테이블을 통해 메모리를 참조함
  2. 트랩(Trap) 발생: 참조한 페이지가 Invalid 상태면 페이지 폴트 트랩이 발생해 제어권이 OS로 넘어감
  3. 디스크 탐색: OS는 디스크에서 해당 프로세스의 페이지를 찾음
  4. 페이지 로드: 빈 프레임이 있다면 해당 프레임에 페이지를 올림
    (빈 프레임이 없다면 페이지 교체 알고리즘(LRU, FIFO 등) 을 통해 기존 페이지를 쫓아냄)
  5. 테이블 갱신: 페이지 테이블에 물리 주소를 기록하고 상태를 Valid로 변경
  6. 명령 재시작: 인터럽트되었던 명령을 다시 실행함

4. 스레싱 (Thrashing)

물리 메모리에 올라간 다중 프로그래밍의 정도(Degree of Multiprogramming)가 너무 높아져서,
프로세스들이 실행보다 페이지 교체(Swap-in/out)에 더 많은 시간을 소비하여 CPU 이용률이 급격히 떨어지는 현상

  • 발생 원인:
    • 메모리에 올라간 프로세스 수가 너무 많아 각 프로세스에 할당된 프레임 수가 절대적으로 부족할 때 발생
    • 페이지 폴트가 빈번하게 발생하여 CPU는 쉬고 디스크 I/O 작업만 계속됨

스레싱 예방 및 해결 방안

  1. 워킹 세트 (Working Set) 모델
    • 프로세스가 일정 시간 동안 자주 참조하는 페이지들의 집합(워킹 세트)을 메모리에 한꺼번에 올려둠
    • 지역성(Locality) 을 기반으로 하여 프레임에 고정시키므로 빈번한 페이지 폴트를 방지함
  2. PFF (Page-Fault Frequency) 방식
    • 프로세스별 페이지 폴트 빈도의 상한선과 하한선을 설정
    • 상한선을 초과하면 빈 프레임을 더 할당하고, 하한선 미만이면 할당된 프레임을 회수하여 적절한 프레임 수를 유지함
