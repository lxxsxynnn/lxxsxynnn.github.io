2026-06-23

1. 요약

배열(Array)

  • 정해진 크기만큼 데이터가 일렬로 저장되는 정적 자료구조
  • 각 데이터를 요소(element), 데이터를 가리키는 번호를 인덱스(index) 라고 함
  • 일반적으로 배열의 인덱스는 0 base

배열의 시간 복잡도

연산 복잡도 비고
접근 O(1) 인덱스로 바로 접근
검색 O(n) 전체 순회 필요
삽입 O(n) 단, 가용 공간이 있어 마지막 인덱스 +1에 삽입하는 경우 O(1)
삭제 O(n) 단, 마지막 요소 삭제 시 O(1)

연결 리스트(Linked List)

  • 크기가 정해져 있지 않은 동적 자료구조
  • 여러 개의 노드(데이터 + 다음 노드의 주소 값)로 구성됨
  • 헤드 포인터(첫 번째 노드를 가리킴)와 테일 포인터(다음 노드 주소 값이 NULL)로 시작과 끝을 알 수 있음
  • 노드가 메모리에 연속적으로 저장되지 않아도 연산을 수행할 수 있음
    • → 데이터의 추가 및 삭제가 자유로움
    • 인덱스가 없어 특정 위치 데이터 접근 시 배열보다 시간이 오래 걸림
head                                    tail
 │                                       │
 ▼                                       ▼
┌──────┬───┐   ┌──────┬───┐   ┌──────┬──────┐
│ data │ ●─┼──▶│ data │ ●─┼──▶│ data │ NULL │
└──────┴───┘   └──────┴───┘   └──────┴──────┘
  노드 1          노드 2          노드 3

연결 리스트의 시간 복잡도

연산 복잡도 비고
검색 O(n) 처음부터 순차 탐색
추가 O(n) 삽입 자체는 O(1)이나, 추가할 위치까지 이동에 O(n) 소요
삭제 O(n) 첫 번째 데이터 삭제는 O(1), 나머지는 이동 후 삭제하므로 O(n)

cf) 연결 리스트의 종류

  1. 이중 연결 리스트

    • 앞·뒤 노드의 주소 값을 모두 저장 → 양방향 탐색 가능
    • 노드를 연속적으로 탐색해야 할 때 단순 연결 리스트보다 시간 면에서 효율적
    • 대신 구현이 더 어렵고, 한 노드당 주소 값 2개를 저장해 메모리를 많이 차지함
            ┌───┬──────┬───┐   ┌───┬──────┬───┐   ┌───┬──────┬───┐
NULL ◀──┼─● │ data │ ●─┼──▶┼─● │ data │ ●─┼──▶┼─● │ data │ ●─┼──▶ NULL
        └───┴──────┴───┘◀──┴───┴──────┴───┘◀──┴───┴──────┴───┘
          prev      next

  2. 원형 연결 리스트

    • 마지막 노드가 NULL이 아니라 첫 번째 노드의 주소 값을 가리킴
    • 새 노드를 맨 앞·맨 뒤에 삽입할 때 상수 시간 소요 → 삽입·삭제가 효율적
    • 순환 구조라 어느 노드에서든 다른 모든 노드에 접근 가능
         ┌──────────────────────────────────────────┐
     │                                          │
     ▼                                          │
┌──────┬───┐   ┌──────┬───┐   ┌──────┬───┐      │
│ data │ ●─┼──▶│ data │ ●─┼──▶│ data │ ●─┼──────┘
└──────┴───┘   └──────┴───┘   └──────┴───┘

스택(Stack)

  • 데이터를 쌓는 형태, 후입선출(LIFO) 구조
  • push: 데이터 삽입 (가장 위에 저장) / pop: 마지막 저장 데이터 삭제
  • top 변수로 마지막 저장된 데이터의 인덱스를 기억함
  • 배열과 연결 리스트로 구현
  • 모든 핵심 연산이 O(1): push, pop, peek, isEmpty, isFull
  • 활용: 작업 실행 취소(undo), 브라우저 뒤로 가기 등 최근에 처리한 작업을 하나씩 꺼낼 때
       push ↓   ↑ pop
        ┌─────────┐
  top → │    30   │
        ├─────────┤
        │    20   │
        ├─────────┤
        │    10   │
        └─────────┘
   (마지막에 넣은 30이 가장 먼저 나옴 — LIFO)

큐(Queue)

  • 데이터가 순차적으로 들어오는 형태, 선입선출(FIFO) 구조
  • 맨 앞을 front, 맨 뒤를 rear라고 하며 이를 이용해 삽입·삭제 연산의 시간 복잡도를 줄임
  • 인큐(enqueue): rear에 데이터 삽입 / 디큐(dequeue): front에서 데이터 삭제
  • 배열과 연결 리스트로 구현
  • 핵심 연산이 O(1): enqueue, dequeue, peek, isEmpty, isFull
  • 활용: 프로세스가 할당받기 전 대기하는 준비 큐처럼, 요청이 들어온 순서대로 처리할 때
dequeue ◀──┐                      ┌──◀ enqueue
           │                      │
     front ▼                      ▼ rear
        ┌──────┬──────┬──────┬──────┐
        │  10  │  20  │  30  │  40  │
        └──────┴──────┴──────┴──────┘
      (먼저 들어온 10이 가장 먼저 나옴 — FIFO)

cf) 순환 큐(원형 큐)

  • 큐를 배열로 구현하면 rear 인덱스로 가득 찼는지 쉽게 확인 가능하고, 인큐·디큐 시 front나 rear만 수정하면 됨
  • 다만 큐를 꽉 채운 뒤 1개만 남기고 삭제하면, rear 인덱스만 보고 가득 찼다고 오판할 수 있음
    • 데이터 위치를 옮겨 해결하면 시간 복잡도가 늘어남
    • → 이를 해결하기 위해 순환 큐(원형 큐) 등장: 배열의 앞부분으로 순환하며 삽입해 시작과 끝의 구분이 사라짐

cf) 덱(Deque, Double-Ended Queue)

  • 양쪽 끝에서 데이터의 삽입·삭제가 모두 가능한 자료구조 (큐 + 스택을 합친 형태)
  • front/rear 양쪽에서 삽입·삭제가 가능하므로 큐로도, 스택으로도 동작시킬 수 있음
    삽입/삭제 ⇄ ┌──────┬──────┬──────┬──────┐ ⇄ 삽입/삭제
     (front)  │  10  │  20  │  30  │  40  │  (rear)
              └──────┴──────┴──────┴──────┘

덱 vs 양방향(이중) 연결 리스트의 차이점

구분 덱(Deque) 양방향 연결 리스트
성격 ADT(추상 자료형) — “무엇을 할 수 있는가"를 정의 자료구조 — “어떻게 저장하는가"를 정의
연산 범위 양쪽 끝(front/rear)에서의 삽입·삭제만 허용 중간 노드 포함 임의 위치 삽입·삭제·탐색 가능
구현 방식 배열 또는 (양방향) 연결 리스트로 구현됨 그 자체가 노드 연결로 이루어진 구현체
  • 즉, 덱은 “양 끝에서만 넣고 빼는” 동작 규칙을 정의한 개념이고, 양방향 연결 리스트는 그 덱을 구현하는 수단 중 하나
  • 양방향 연결 리스트는 양 끝뿐 아니라 중간 어디에서든 노드를 추가·삭제할 수 있다는 점에서 덱보다 자유로움
