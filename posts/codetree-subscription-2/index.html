코딩테스트를 준비하면서 항상 막연하게 느꼈던 게 있었습니다.

“문제는 이해했는데 왜 구현 단계만 가면 자꾸 꼬일까?”

이번에 코드트리의 갭체크와 시뮬레이션 유형 학습을 진행하면서 그 이유를 꽤 명확하게 알 수 있었습니다.
단순히 문법이나 알고리즘 개념을 아는 것과, 실제로 조건을 빠짐없이 구현하는 건 완전히 다른 문제였습니다.

시뮬레이션 문제를 풀며 가장 크게 느낀 점

이번에 공부한 유형은 시뮬레이션(Simulation) 이었습니다.

처음에는 단순 구현 문제라고 생각했는데, 실제로 풀어보니 조건 하나만 빠져도 결과가 완전히 달라지는 유형이라 생각보다 훨씬 어려웠습니다.

특히 코드트리에서 좋았던 점은 문제만 던져주는 게 아니라,
“왜 이런 방식으로 접근해야 하는지"를 먼저 설명해준다는 점이었습니다.

예를 들어 시간 계산 문제에서는 현재 시간을 1분씩 증가시키며 시뮬레이션하는 방법부터 설명해준 뒤,

while(true) {
    if(hour == 4 && mins == 1)
        break;

    elapsedTime++;
    mins++;

    if(mins == 60) {
        hour++;
        mins = 0;
    }
}

이후에는 더 효율적인 사고 방식으로 확장했습니다.

“0시 0분부터 특정 시각까지 몇 분이 흘렀는가?“를 계산해서 두 시간의 차이를 구하는 방식으로 접근을 바꾸는 식이었습니다.
처음에는 어떻게 접근해야 하나 막막했는데, 이런 식으로 문제를 더 단순한 수학적 형태로 바꾸는 사고 과정을 보면서 많이 배웠습니다.

점진적으로 확장되는 학습 방식

특히 인상 깊었던 건 코드트리의 학습 흐름이었습니다.

처음에는 정말 기초적인 구현부터 시작합니다.

  • 시간 증가
  • 좌표 이동
  • 배열 순회

이런 간단한 문제를 먼저 풀게 하고, 이후에는 점점 조건을 추가하면서 난이도를 올려갑니다.

예를 들면:

  • 원형 배열 처리
  • 시계/반시계 방향 이동
  • 범위를 벗어나는 경우 처리

이런 식으로 확장되는데, “이전 문제의 개념을 응용해서 다음 문제를 푸는 구조"라 공부 흐름이 굉장히 자연스러웠습니다.
덕분에 구현 패턴 자체를 익히는 데 도움이 많이 됐습니다.

아직 부족하다고 느낀 부분

이번 학습을 하면서 특히 부족하다고 느낀 건 예외 처리 경험이었습니다.

특히 자주 실수하는 부분이 있었는데,

  • 인덱스 범위 초과
  • 오프셋(offset) 계산 실수
  • 시작 위치 포함 여부
  • +1, -1 처리
  • 원형 배열에서 % n 처리

문제를 풀면서 로직 자체는 맞지만 경계 조건 때문에 오답이 계속 발생했습니다.

원형 배열 문제에서도 단순히 % n만 사용하면 될 줄 알았는데,
아래와 같이 음수 방향 이동까지 고려해야 해야 하는 부분을 간과했었습니다.

start = (start + direction + n) % n;

특히 배열을 사용할 때는 음의 수 처리나 배열 범위 초과 오류가 나지 않게 주의를 기울여야겠다는 생각을 했고,
알고리즘을 많이 구현해보면서 몸으로 익혀야 한다는 걸 느꼈습니다.

갭체크를 통해 느낀 내 약점

갭체크 결과를 보면서 가장 공감됐던 부분은
“구현은 가능하지만 안정성이 부족하다"는 분석이었습니다.

문제를 읽고 아이디어를 떠올리는 건 어느 정도 가능했지만,

  • 조건 누락
  • 예외 상황 처리 부족
  • 구현 중 인덱스 꼬임
  • 디버깅 시간 증가

같은 문제로 인해 완성도가 떨어졌습니다.

작은 실수 하나가 전체 결과에 영향을 주기 때문에, 알고리즘 지식도 중요하지만 꼼꼼함과 구현 경험 역시 중요하다는 걸 체감했습니다.

앞으로의 공부 방향

당분간은 시뮬레이션 유형을 반복해서 풀면서 구현 안정성을 높이는 데 집중하려고 합니다.

다음 부분을 중심으로 많이 연습할 생각입니다.

  1. 인덱스/오프셋 처리 익숙해지기
  2. 원형 배열과 방향 처리 반복 연습
  3. 조건을 빠짐없이 체크하는 습관 만들기
  4. 구현 전에 흐름을 먼저 그림으로 정리하기

무작정 문제 수를 늘리기보다,
“왜 여기서 틀렸는지"를 정확히 분석하면서 푸는 게 훨씬 중요하다는 걸 느꼈습니다.

마무리

이번 코드트리 학습을 통해 어떤 방식으로 사고해야 하는지를 배우는 게 정말 중요하다는 걸 느꼈습니다.
특히 시뮬레이션 유형은 작은 예외 처리 하나가 정답 여부를 결정하기 때문에, 구현 경험이 가장 중요한 것 같습니다.
아직 인덱스 처리나 경계 조건 같은 부분이 많이 부족하지만, 반복 학습을 통해 조금씩 안정적으로 구현할 수 있도록 계속 연습해보려고 합니다.
