코딩테스트 공부를 하다 보면 “이건 DFS다”, “백트래킹이다” 하는 말을 정말 자주 듣습니다.

이번 회차에서는 DFS / 백트래킹을 제 방식대로 정리해보려고 합니다.
“그래서 둘이 뭐가 다른데?“부터 시작해서, 구현 템플릿과 실수하기 쉬운 함정까지 한번에 짚고 넘어갈게요.

👉 코드트리 바로가기
코드트리(Codetree)

1. DFS란 무엇인가

DFS(Depth-First Search, 깊이 우선 탐색)는 이름 그대로 한 길을 끝까지 파고든 뒤, 막다른 길을 만나면 다시 돌아와서 다른 길을 시도하는 탐색 방식입니다.

말로 설명하면 어려운데, 미로 게임을 떠올리면 쉽습니다.

“갈 수 있는 길이 보이면 일단 들어간다.
막다른 길이 나오면 한 칸 뒤로 돌아온다.
모든 방향을 다 시도해봤다면 또 한 칸 뒤로 돌아온다.”

이 동작을 코드로 옮긴 게 DFS입니다. 그래프, 트리, 2차원 격자 — “노드와 그 옆 노드들"로 표현할 수 있는 모든 구조에 적용할 수 있습니다.

DFS의 두 가지 구현 방식

구현 방식 특징
재귀 호출 코드가 짧고 직관적. 호출 스택을 그대로 활용
명시적 스택 재귀 깊이가 너무 깊어 StackOverflow가 우려될 때 사용

대부분의 코딩테스트 문제에서는 재귀 방식으로 충분합니다.

2. 백트래킹은 DFS와 뭐가 다를까

처음 공부할 때 제일 헷갈렸던 부분입니다. 결론부터 말하면, **백트래킹은 DFS의 “전략적인 버전”**입니다.

  • DFS: 모든 경로를 다 탐색한다.
  • 백트래킹: 탐색하되, “답이 될 수 없는 가지"는 미리 잘라낸다 (가지치기, Pruning).

예를 들어 N-Queens 문제에서 첫 번째 행에 퀸을 두고, 두 번째 행에 두려고 하는데 이미 같은 열에 있는 경우를 생각해봅시다.
DFS는 그래도 일단 끝까지 들어가 봤다가 “아 안 되네” 하고 돌아오겠죠. 백트래킹은 두 번째 행에서 이미 불가능을 알았기 때문에, 그 가지로 아예 안 들어갑니다.

즉 백트래킹의 핵심은 두 가지입니다.

  1. 유망성 판단(Promising Check): 이 길로 가면 답이 나올 수 있는가?
  2. 상태 복구(Backtrack): 다음 길을 시도하기 위해 왔던 흔적을 지운다.

이 두 가지가 빠지면 백트래킹이 아니라 그냥 완전 탐색 DFS입니다.

3. 구현 패턴

DFS/백트래킹 문제를 풀 때 항상 다음 세 가지를 먼저 그립니다.

  1. 종료 조건: 언제 재귀를 멈출 것인가?
  2. 다음 후보: 현재 상태에서 갈 수 있는 곳은 어디인가?
  3. 상태 복구: 다음 가지를 시도하기 전에 무엇을 되돌려야 하는가?

이걸 코드로 옮기면 거의 항상 이런 모양이 됩니다.

void dfs(int node, /* 현재까지의 상태 */) {
    // 1. 종료 조건
    if (목표_달성(node)) {
        정답_처리();
        return;
    }

    // 2. 다음 후보 탐색
    for (int next : 다음_후보(node)) {
        if (visited[next]) continue;         // 이미 방문한 곳
        if (!유망한가(next)) continue;        // 백트래킹: 가지치기

        visited[next] = true;                // 상태 변경
        dfs(next, /* 갱신된 상태 */);
        visited[next] = false;               // 상태 복구 (백트래킹의 핵심!)
    }
}

격자 문제라면 다음_후보가 상하좌우 4방향이 되고, 순열 생성이라면 “아직 안 쓴 숫자들"이 됩니다.
틀만 같고 안에 채워 넣는 게 달라질 뿐이라는 게 DFS/백트래킹의 매력입니다.

격자 문제의 단골 템플릿 

int[] dx = {-1, 1, 0, 0};
int[] dy = {0, 0, -1, 1};

void dfs(int x, int y) {
    visited[x][y] = true;

    for (int dir = 0; dir < 4; dir++) {
        int nx = x + dx[dir];
        int ny = y + dy[dir];

        if (nx < 0 || ny < 0 || nx >= N || ny >= M) continue; // 격자 밖
        if (visited[nx][ny]) continue;
        if (board[nx][ny] == 0) continue;                     // 갈 수 없는 칸

        dfs(nx, ny);
    }
}

4. 코드트리 문제 풀이로 익혀보기

직접 풀어봐야 이해가 더 잘 갈 것 같아서 코드트리 그래프 탐색 트레일의 입문 문제 **“DFS 행렬”**을 풀면서 더 자세히 알아보겠습니다.

👉 문제 보러 가기
코드트리 - 그래프 탐색 입문 트레일

문제 요약

5개의 도시가 있고, 도시 간 연결 관계가 5×5 인접 행렬로 주어집니다. (a[i][j] = 1이면 i → j 간선 존재)

     1   2   3   4   5
  1  0   0   1   0   1
  2  1   0   0   1   0
  3  1   0   0   0   0
  4  0   1   0   0   1
  5  0   0   0   1   0

1번 도시에서 출발해 DFS를 수행하되, 갈 수 있는 도시가 여러 개일 때는 번호가 작은 도시부터 탐색합니다. 이때의 방문 순서를 구하면 됩니다.

접근 방법

  1. 인접 행렬 + visited 배열: 정점 수가 적으니 인접 행렬을 그대로 사용
  2. “작은 번호부터” 조건: 인접 행렬의 열 번호를 1부터 N까지 오름차순으로 순회하면 자동으로 만족됩니다. 별도의 정렬이 필요 없습니다.

핵심 로직 

static int N = 5;
static int[][] graph = new int[N + 1][N + 1];
static boolean[] visited = new boolean[N + 1];
static List<Integer> order = new ArrayList<>();

static void dfs(int node) {
    visited[node] = true;
    order.add(node);

    // 작은 번호부터 보기 위해 1부터 순회
    for (int next = 1; next <= N; next++) {
        if (graph[node][next] == 1 && !visited[next]) {
            dfs(next);
        }
    }
}

public static void main(String[] args) {
    // 입력 파싱 ...
    dfs(1);
    // order: [1, 3, 5, 4, 2]
}

손으로 따라가 보기

단계 현재 노드 visited 갈 수 있는 후보 선택
① 1 {1} 3, 5 3
② 3 {1, 3} 1(방문) ← 백트랙
③ 1 {1, 3} 5 5
④ 5 {1, 3, 5} 4 4
⑤ 4 {1, 3, 5, 4} 2, 5(방문) 2
⑥ 2 {1, 3, 5, 4, 2} 1(방문), 4(방문) ← 종료

방문 순서: 1 → 3 → 5 → 4 → 2

표로 정리하니 “왜 5보다 3을 먼저 갔는지”, “왜 2가 가장 마지막인지"가 한눈에 보입니다. 트레이싱을 손으로 한번 해보는 게 사실 가장 빠른 학습법이에요.

시간 복잡도

  • 인접 행렬을 모든 정점에서 한 번씩 훑으므로 O(V²).
  • 정점이 적은 (V ≤ 1,000 정도) 문제에서는 인접 행렬이 오히려 구현이 깔끔합니다. 정점이 많아지면 인접 리스트로 바꿔 O(V + E)로 줄이는 게 정석입니다.

한 단계 더 — 격자에서의 DFS

같은 DFS 골격을 격자 문제에 적용하면 또 다른 결의 문제를 풀 수 있습니다.
예를 들어 코드트리의 뿌요뿌요 유형 문제처럼 같은 색이 4개 이상 인접해 있는지 찾는 문제도 결국 “특정 칸에서 출발해 같은 색으로 이어진 칸을 모두 세는 DFS"입니다.

5. 자주 하는 실수와 함정

제가 직접 겪었거나, 코드트리 문제를 풀면서 가장 많이 본 실수들을 모아봤습니다.

① 상태 복구를 빼먹는다 

visited[next] = true;
dfs(next);
// visited[next] = false;  ← 이걸 까먹는 순간 답이 한 갈래로만 나옴

특히 “모든 경우의 수"를 구하는 문제에서 이게 빠지면 첫 번째로 도달한 답만 나오고 나머지 가지를 못 타게 됩니다.

② 종료 조건 위치를 잘못 잡는다

종료 조건을 재귀 호출 후에 두면 이미 한 단계 더 들어간 상태에서 검사하게 됩니다.
항상 함수 진입 직후 맨 처음에 두는 것을 습관화하는 게 좋을 것 같습니다.

③ 격자 문제에서 인덱스 범위 검사를 빠뜨린다 

if (nx < 0 || ny < 0 || nx >= N || ny >= M) continue;

이 한 줄 없으면 ArrayIndexOutOfBoundsException이 발생하거나, 더 골치 아프게는 “잘못된 위치"를 방문 처리해서 결과가 미묘하게 어긋납니다.

④ 가지치기를 너무 일찍 한다

백트래킹이 좋다고 무리하게 가지치기 조건을 넣다 보면, 답이 될 수 있는 경로까지 잘라버리는 경우가 있습니다.
가지치기 조건을 짤 때는 “이 조건이 참이면 절대로 답이 안 나오는가?“를 한 번 더 확인하세요.

⑤ 재귀 깊이가 너무 깊다

Java의 기본 스택 크기는 보통 512KB ~ 1MB 정도입니다. 재귀 호출이 수만 번 일어나는 문제라면 명시적 스택으로 바꾸거나, -Xss 옵션으로 스택을 키우는 걸 고려해야 합니다.

6. 기타 — 풀이를 빠르게 만드는 사고 순서

문제를 받았을 때 저는 이 순서로 생각합니다.

  1. “그래프/트리/격자 형태인가?” → DFS 후보
  2. “모든 경우의 수를 다 봐야 하는가?” → 백트래킹
  3. “중간에 잘라낼 수 있는 명백한 조건이 있는가?” → 있으면 가지치기 설계
  4. “상태가 무엇인가?” → 재귀 인자로 들고 다닐 변수 결정
  5. “상태를 어떻게 복구하는가?” → 백트래킹 라인 설계

이 다섯 가지 질문에 답이 나오면, 사실상 코드의 골격은 다 잡힌 셈입니다.

💡 TIP: 상태 복구가 너무 복잡해지면 그 문제는 DFS보다 BFS가 더 적합할 수도 있습니다.
“왔던 길을 되돌릴 수 있는가?“가 DFS와 BFS를 가르는 중요한 기준입니다.

마치며

처음에는 “DFS랑 백트래킹은 그냥 같은 거 아닌가?“라고 생각했는데, 코드트리에서 다양한 유형을 풀어보면서 둘의 결정적인 차이가 가지치기와 상태 복구라는 걸 체감했습니다.
특히 격자형 시뮬레이션이나 부분집합/순열 문제에서 이 두 가지가 빠지면 시간 초과로 직행하거나, 답이 한 개밖에 안 나오는 황당한 일이 벌어지더라고요.

이번 정리를 통해 제 머릿속에서도 한번 더 구조가 정돈된 느낌입니다.
앞으로 비슷한 유형을 만났을 때, 이 글에 적어둔 “종료 조건 → 다음 후보 → 상태 복구” 세 줄짜리 체크리스트를 떠올리면서 풀어볼 생각입니다.
