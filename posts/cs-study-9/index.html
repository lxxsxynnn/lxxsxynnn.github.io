[CS 스터디] TCP와 UDP

2026-06-08

1. TCP (Transmission Control Protocol)

전송 계층(L4)에 속하는 프로토콜로, 연결형 서비스와 데이터의 신뢰성을 보장함

쉽게 이해하기: TCP는 “등기우편"과 같음. 상대방이 받았는지 확인하고, 순서대로 전달되며, 분실되면 다시 보냄. 대신 느림.

  • 주요 특징:
    • 송신부와 수신부의 연결을 먼저 확인하는 연결형 서비스
    • 가상 회선 방식의 패킷 교환 (모든 패킷이 같은 경로로 이동)
    • 패킷의 전송 순서 보장 및 수신 여부 확인
    • 1:1 통신만 가능
    • 데이터 손실이 없어 신뢰성이 높음
    • 절차가 많아 속도가 느림

패킷 교환 방식 비교 

   [ 가상 회선 방식 (TCP) ]              [ 데이터그램 방식 (UDP) ]

   송신부 → [P1][P2][P3] → 수신부          송신부 → [P1] → 경로A → 수신부
                                                → [P2] → 경로B → 수신부
        (모두 같은 경로)                          → [P3] → 경로C → 수신부
        (순서 보장)                              (순서가 뒤바뀔 수 있음)

2. TCP 핸드셰이킹 (Handshaking)

연결형 서비스를 위해 송신부와 수신부가 연결을 맺고 끊는 과정

쉽게 이해하기: 전화 통화를 시작할 때 “여보세요?” → “네 잘 들려요” → “좋아요 얘기 시작합시다"처럼 서로 연결을 확인하는 절차

주요 플래그(Flag)

플래그 의미
SYN 연결 생성 요청 (Synchronize)
ACK 데이터 수신 확인 (Acknowledge)
FIN 연결 종료 요청 (Finish)
RST 연결 재설정 (Reset)
PSH 빠른 응답이 필요한 데이터 즉시 전송 (Push)
URG 우선순위가 높은 긴급 데이터 전송 (Urgent)

3-way Handshake (연결 수립) 

   [ 송신부 (Client) ]                          [ 수신부 (Server) ]

         │                                              │
   ① ────│──── SYN, seq=N ────────────────────────────>│
   SYN_SENT                                             │
         │                                              │ SYN_RECEIVED
   ②    │<──── SYN + ACK, seq=M, ack=N+1 ──────────────│
         │                                              │
   ESTABLISHED                                          │
   ③ ────│──── ACK, ack=M+1 ────────────────────────>│
         │                                              │ ESTABLISHED
         │           ===== 연결 완료 =====              │
  1. 송신부 → 수신부: 연결 요청(SYN)과 임의의 숫자 N을 보냄 → SYN_SENT 상태
  2. 수신부 → 송신부: 요청 수락(SYN + ACK)과 N+1, 임의의 숫자 M을 보냄 → SYN_RECEIVED 상태
  3. 송신부 → 수신부: 수락 확인(ACK)과 M+1을 보냄 → 양쪽 모두 ESTABLISHED 상태

4-way Handshake (연결 종료) 

   [ 송신부 (Client) ]                          [ 수신부 (Server) ]

         │                                              │
   ① ────│──── FIN ───────────────────────────────────>│
   FIN_WAIT1                                            │
         │                                              │ CLOSE_WAIT
   ②    │<──── ACK ───────────────────────────────────│ (종료 준비)
   FIN_WAIT2                                            │
         │                                              │
   ③    │<──── FIN ───────────────────────────────────│ LAST_ACK
         │                                              │
   TIME_WAIT                                            │
   ④ ────│──── ACK ───────────────────────────────────>│
         │                                              │ CLOSED
   (일정 시간 대기 후)                                   │
   CLOSED                                               │
  1. 송신부 → 수신부: 연결 종료 요청(FIN) → FIN_WAIT1 상태
  2. 수신부 → 송신부: 응답(ACK) → 수신부는 CLOSE_WAIT, 송신부는 FIN_WAIT2 상태
  3. 수신부 → 송신부: 종료 준비 완료 후 FIN 전송 → LAST_ACK 상태
  4. 송신부 → 수신부: 응답(ACK) → TIME_WAIT 상태, 일정 시간 후 CLOSED

TIME_WAIT을 두는 이유: 마지막 ACK가 유실되거나 늦게 도착하는 패킷에 대비하기 위함

3. TCP 제어 방법

① 흐름 제어 (Flow Control)

송신부와 수신부의 처리 속도 차이로 발생하는 데이터 손실을 방지

쉽게 이해하기: 친구에게 공을 던지는데, 친구가 받는 속도보다 빠르게 던지면 공이 떨어짐. 친구가 받을 수 있는 속도에 맞춰서 던지는 것

  • 정지-대기 (Stop-and-Wait): 데이터 1개를 보내고 ACK를 받은 후에야 다음 데이터를 보냄 → 간단하지만 매우 비효율적

  • 슬라이딩 윈도우 (Sliding Window): 수신부가 정한 윈도우 크기만큼 ACK를 기다리지 않고 연속 전송 → 동적으로 데이터 흐름 제어 가능 → 윈도우 크기는 3-way 핸드셰이킹 시 결정됨

   [ 슬라이딩 윈도우 동작 ]

   윈도우 크기 = 4

   [P1][P2][P3][P4] P5  P6  P7  P8       ← 처음
    └──── 전송 ────┘

   P1 [P2][P3][P4][P5] P6  P7  P8       ← P1의 ACK 도착 시 윈도우 이동
       └──── 전송 ────┘

   P1  P2 [P3][P4][P5][P6] P7  P8       ← P2의 ACK 도착 시 윈도우 이동
           └──── 전송 ────┘

② 혼잡 제어 (Congestion Control)

송신부의 전송 속도와 네트워크 속도 차이로 발생하는 데이터 손실을 방지 (혼잡: 네트워크에 패킷이 과도하게 많아진 상태)

쉽게 이해하기: 도로에 차가 너무 많으면 정체가 일어남. 차를 천천히 보내야 하는 상황

  • AIMD (Additive Increase / Multiplicative Decrease):

    • 정상일 땐 윈도우 크기를 1씩 더해가며 증가
    • 손실 발생 시 윈도우 크기를 1/2로 감소

  • 느린 시작 (Slow Start):

    • 윈도우 크기를 1로 시작 → ACK 받을 때마다 2배씩 증가 (지수 함수)
    • 혼잡 발생 시 윈도우 크기를 1로 초기화

  • 혼잡 회피 (Congestion Avoidance):

    • 임계점(ssthresh)에 도달하면 선형 증가로 전환
    • 타임아웃 발생 시 임계점을 현재 윈도우의 1/2로 설정 후 초기화

  • 빠른 회복 (Fast Recovery):

    • 혼잡 시 윈도우를 절반으로 줄인 뒤 선형 증가
    • 첫 혼잡 시에는 AIMD 방식으로 동작

  • 빠른 재전송 (Fast Retransmit):

    • Duplicate ACK가 3번 발생하면 즉시 재전송
    • 윈도우 크기를 1/2로 줄임

TCP Tahoe vs TCP Reno 

   [ TCP Tahoe ]                          [ TCP Reno ]

   윈도우 크기                              윈도우 크기
     │      /\                              │      /\        /\
   6 │     /  \                           6 │     /  \      /  \
     │    /    \                            │    /    \    /
   4 │ ssthresh \                         4 │ ssthresh \  / ssthresh 변경
     │  /        \                          │  /        \/
   2 │ /          \    /\                 2 │ /
     │/            \  /  \                  │/
   1 │              \/    \               1 │
     └─────────────────────→ 시간            └─────────────────────→ 시간

   * 혼잡 발생 시 → 윈도우 1로 초기화        * 3 Dup ACK → 윈도우 1/2, 선형 증가
   * ssthresh = 혼잡 시점의 1/2              * 타임아웃 → 윈도우 1로 초기화
                                              (ssthresh는 유지)
  • TCP Tahoe: 모든 혼잡(Timeout, 3 Dup ACK)에 대해 윈도우 크기를 1로 초기화
  • TCP Reno: 3 Duplicate ACK와 타임아웃을 구분해서 대응
    • 3 Dup ACK → 윈도우 1/2 후 선형 증가 (빠른 회복)
    • 타임아웃 → 윈도우 1로 초기화 (ssthresh는 유지)

③ 오류 제어 (Error Control)

통신 중 오류·유실 발생 시 신뢰성 보장을 위한 제어

  • 오류를 감지하는 방법:

    • 수신부가 잘못된 데이터를 받았다는 NAK 메시지
    • 3 Duplicate ACK 발생
    • ACK를 받지 못해 타임아웃 발생

  • 재전송 방식 (ARQ: Automatic Repeat reQuest):

   [ Go-Back-N ARQ ]                      [ Selective-Repeat ARQ ]

   송신: [0][1][2][3][4][5]                송신: [0][1][2][3][4][5]
              ✗ (2번 유실)                          ✗ (2번 유실)
   수신: [0][1] ──── ACK 2 ────>           수신: [0][1][_][3][4][5]
                                                 └─ ACK 2만 요청 ─┐
   재전송: [2][3][4][5] ←── 모두           재전송: [2] ←── 2번만
            다시 보냄                                다시 보냄

   * 단순하지만 비효율                       * 효율적이지만 재정렬 필요
  • 정지-대기: 타임아웃 발생 시 해당 패킷 재전송
  • Go-Back-N ARQ: 누락된 패킷부터 이후 모든 패킷 재전송
  • Selective-Repeat ARQ: 누락된 패킷만 선택적으로 재전송 (재정렬 로직 필요)

4. UDP (User Datagram Protocol)

전송 계층(L4)에 속하는 프로토콜로, 비연결형 서비스와 빠른 전송을 지원

쉽게 이해하기: UDP는 “전단지 뿌리기"와 같음. 받는 사람이 받았는지 신경 쓰지 않고 빠르게 던짐. 영상 스트리밍, 게임처럼 약간의 손실보다 속도가 중요한 곳에 사용

  • 주요 특징:
    • 연결 과정 없는 비연결형 서비스
    • 데이터그램 방식 (패킷마다 다른 경로 가능)
    • 패킷의 전송 순서 보장 X
    • 수신 여부 확인 X
    • 1:1, 1:N, N:N 통신 모두 가능
    • 신뢰성은 낮지만 속도가 빠름

5. TCP vs UDP 비교

구분 TCP UDP
연결 방식 연결형 (3-way 핸드셰이크) 비연결형
패킷 교환 방식 가상 회선 방식 데이터그램 방식
전송 순서 보장 보장하지 않음
수신 여부 확인 O X
통신 방식 1:1 1:1, 1:N, N:N
신뢰성 높음 낮음
속도 느림 빠름
사용 예시 웹(HTTP), 이메일, 파일 전송 영상 스트리밍, 온라인 게임, DNS

6. UDP의 오류 검출 (체크섬)

UDP는 신뢰성이 낮지만, 체크섬(Checksum) 방식으로 최소한의 오류 검출 가능

쉽게 이해하기: 영수증의 합계 금액과 같음. 항목들을 더한 값이 일치하지 않으면 뭔가 잘못된 것

체크섬 동작 과정 

   [ 송신부 ]                                    [ 수신부 ]

   ① UDP 헤더 + IP 정보 + 데이터               ① 받은 모든 값 + 체크섬을
      를 모두 더함                                   다시 더함
                                                ↓
   ② 캐리(오버플로우) 발생 시                   ② 비트가 모두 1이면
      캐리를 떼서 다시 더함                          → 정상
                                                    아니면 → 오류
   ③ 1의 보수를 취해 체크섬 생성

   ④ UDP 헤더의 체크섬 영역에 넣어 전송   ───────>

  • 체크섬 생성:

    1. UDP 헤더 + IP 헤더 일부(발신/수신 IP, 프로토콜 ID) + 데이터를 모두 더함
    2. 캐리(오버플로우) 발생 시 떼어내 다시 더함
    3. 1의 보수를 취해 체크섬 생성

  • 체크섬 검증:

    • 수신부는 체크섬 포함 모든 값을 더해 비트가 모두 1이 나오는지 확인
    • 모두 1이면 정상, 아니면 오류

  • 한계점:

    • 데이터를 단순히 더해 비교하므로 100% 검출 불가
    • 순서가 바뀌어도 합이 같으면 검출 못 함
    • UDP의 체크섬은 선택 사항 (값이 0이면 검증 안 함)

7. 주요 용어 정리

  • 세그먼트 (Segment): TCP에서 사용하는 데이터 전송 단위
  • 데이터그램 (Datagram): UDP에서 사용하는 독립적인 데이터 단위
  • 핸드셰이킹 (Handshaking): 통신을 시작·종료할 때 양쪽이 신호를 주고받아 연결을 제어하는 과정
  • 윈도우 크기 (Window Size): ACK 없이도 연속해서 보낼 수 있는 데이터의 최대 개수
  • ssthresh (Slow Start Threshold): 느린 시작에서 혼잡 회피로 전환되는 임계점
  • ARQ (Automatic Repeat reQuest): 오류 발생 시 자동으로 재전송을 요청하는 방식
  • 체크섬 (Checksum): 데이터의 무결성을 검증하기 위한 간단한 합계 값
